# オンライン会議リアルタイム文字起こしツール 要件定義・設計書

作成日: 2026-07-06
対象: Claude Codeでの実装用ドキュメント

---

## 1. 概要

Zoom等のオンライン会議の音声を、Windowsデスクトップ上でリアルタイムに文字起こしし、会議終了後にtxtファイルとして保存するツール。

### 利用環境(前提)

| 項目 | 内容 |
|---|---|
| OS | Windows 10/11 |
| 音声デバイス | Bluetoothイヤホンマイク(受話・発話とも) |
| 会議アプリ | Zoom / Teams / Google Meet 等(アプリ非依存で動作させる) |
| 言語 | 日本語メイン(英語混在も許容) |
| 機密性 | 業務会議のため、音声・テキストを外部サービスに送信しない(ローカル完結) |

---

## 2. 要件定義

### 2.1 機能要件

| ID | 要件 | 優先度 |
|---|---|---|
| F-01 | 会議相手の音声(PCの再生音)を文字起こしできる | 必須 |
| F-02 | 自分の音声(マイク入力)を文字起こしできる | 必須 |
| F-03 | 文字起こし結果を発話ごとにリアルタイム表示する(遅延目安: 2〜5秒) | 必須 |
| F-04 | 各発話に「タイムスタンプ」と「話者ラベル(自分/相手)」を付与する | 必須 |
| F-05 | 録音停止時にtxtファイルを自動保存する | 必須 |
| F-06 | 録音の開始/停止をワンクリックで行える | 必須 |
| F-07 | 入力デバイス・出力(ループバック)デバイスをUIで選択できる | 必須 |
| F-08 | アプリ異常終了時もそれまでの内容が失われない(逐次追記) | 必須 |
| F-09 | ウィンドウの常に手前表示(always on top)切替 | 推奨 |
| F-10 | 業務用語のヒント辞書(initial_prompt)を設定ファイルで指定できる | 推奨 |
| F-11 | 保存先フォルダ・ファイル名規則の設定 | 推奨 |
| F-12 | Bluetoothデバイス切断時の検知と再接続後のストリーム復帰 | 推奨 |
| F-13 | 文字起こし結果のMarkdown形式同時出力 | 任意 |
| F-14 | 議事録要約(会議後にローカルまたはAPIで生成) | 将来 |

### 2.2 非機能要件

| ID | 要件 |
|---|---|
| N-01 | 音声データ・テキストはローカルにのみ保存。外部送信なし |
| N-02 | Zoom等と同時稼働してもPCが実用に耐える負荷(CPU使用率の上限を意識したモデル選定) |
| N-03 | 会議1〜2時間の連続稼働に耐える(メモリリークなし) |
| N-04 | 依存ライブラリはpip installで完結し、環境構築が容易 |

### 2.3 スコープ外(v1では作らない)

- 話者分離(相手側の複数人を個別に識別する diarization)→ 「自分/相手」の2区分まで
- クラウド同期、他デバイス連携
- 会議アプリのAPI連携(Zoom SDK等)→ 音声レイヤーで完結させアプリ非依存にする

## 3. 技術選定

### 3.1 全体スタック

| レイヤー | 技術 | 選定理由 |
|---|---|---|
| 言語 | Python 3.11+ | 音声処理・ASRエコシステムが充実 |
| 音声キャプチャ | **PyAudioWPatch** | PyAudioのfork。WASAPIループバック録音に対応(相手音声の取得に必須) |
| 発話区間検出 | **silero-vad** | 軽量・高精度。無音区切りでセグメント化し逐次認識する要 |
| 音声認識 | **faster-whisper** | Whisperの高速実装(CTranslate2)。int8量子化でCPUでも実用。完全ローカル |
| UI | **PySide6** | デスクトップアプリとして安定。常に手前表示等も容易 |
| 設定 | JSON(config.json) | シンプルに |

### 3.2 Whisperモデルの選定方針

| 環境 | 推奨モデル | 備考 |
|---|---|---|
| CPUのみ(まず試す) | `small` (int8) | 日本語の業務会話で実用ライン |
| CPUのみ(本命候補) | `kotoba-tech/kotoba-whisper-v2.0-faster` (int8) | 日本語特化の蒸留モデル。large-v3に近い日本語精度でsmall〜medium並みの速度。faster-whisperからモデル名指定でそのまま利用可 |
| CPUで精度不足時 | `medium` (int8) | Zoom同時稼働では遅延が厳しい可能性。kotoba-whisper優先で検証 |
| NVIDIA GPUあり | `large-v3` (float16/int8_float16) | 精度最良。CUDA環境の場合のみ |

本プロジェクトはCPU環境のため、**M2で `small` と `kotoba-whisper-v2.0-faster` の2つを同一音源で比較**し、精度と処理速度(実時間比)で採用モデルを決定する。

実装上はモデル名をconfig.jsonで切替可能にする。初回起動時にモデルを自動ダウンロード(Hugging Face)。

### 3.3 音声取得の設計(最重要ポイント)

**2系統を並行キャプチャする:**

1. **相手の声** = PCの再生音。WASAPIループバックで「既定の再生デバイス(Bluetoothイヤホン)」の出力をキャプチャする。PyAudioWPatchの `get_default_wasapi_loopback()` を使用。
2. **自分の声** = Bluetoothマイクからの通常入力キャプチャ。

**Bluetoothに関する注意(実装時に必ず考慮):**

- Bluetoothヘッドセットはマイク使用中、A2DP(高音質再生)からHFP(ヘッドセットプロファイル、8〜16kHzモノラル)に切り替わる。会議中は再生音・マイク音とも16kHz相当になるが、**Whisperの入力は16kHzなので文字起こし品質への実害はない**。「音がこもって聞こえる」のはBluetoothの仕様であり本ツールの問題ではない。
- ループバックキャプチャのデバイスは「そのとき音が再生されているデバイス」に一致させる必要がある。会議アプリの出力先とツールのループバック対象がズレると相手音声が拾えないため、UIに「既定デバイスに追従」オプションを設ける。
- Bluetooth切断・再接続時はストリームエラーが発生する。キャプチャスレッドで例外を捕捉し、デバイス再検出→ストリーム再オープンを試みる(F-12)。
- サンプルレートはデバイスのネイティブレートで開き、`numpy`/`scipy`で16kHz mono float32にリサンプルしてASRへ渡す。

## 4. アーキテクチャ

### 4.1 処理パイプライン

```
[マイク入力]────┐
                ├→ キャプチャスレッド×2 → リングバッファ
[ループバック]──┘         │
                          ▼
                 VAD(silero-vad, ch別)
                 発話開始〜無音600msで1セグメント確定
                          │
                          ▼
                 セグメントキュー(queue.Queue)
                 { channel: "self"|"other", audio, start_ts }
                          │
                          ▼
                 ASRワーカースレッド(faster-whisper)
                 ※モデルは1インスタンスを共有、キュー順に処理
                          │
                          ├→ UIへSignal通知(逐次表示)
                          └→ 追記ライター(txtへ逐次append)
```

### 4.2 スレッド構成

| スレッド | 役割 |
|---|---|
| メイン(Qt) | UI描画、Signal/Slot受信 |
| CaptureThread ×2 | マイク/ループバックの読み取り、VAD、セグメント確定 |
| ASRWorker ×1 | キューからセグメントを取り出しfaster-whisperで認識 |
| WriterThread ×1 | 認識結果をファイルへ逐次追記(異常終了対策) |

- スレッド間はすべて `queue.Queue` と Qt Signal で疎結合にする。
- ASRは1スレッドに集約(モデルの多重実行はメモリ・CPU的に不利)。2chのセグメントが同時に来たらキュー順で処理し、表示は `start_ts` でソートして時系列を保つ。

### 4.3 モジュール構成

```
meeting_transcriber/
├── main.py              # エントリポイント
├── config.py            # config.json読み書き
├── audio/
│   ├── devices.py       # デバイス列挙・既定デバイス取得・追従
│   ├── capture.py       # CaptureThread(WASAPI/マイク共通化)
│   └── vad.py           # silero-vadラッパー、セグメンタ
├── asr/
│   └── transcriber.py   # faster-whisperラッパー(ASRWorker)
├── output/
│   └── writer.py        # 逐次追記、停止時の整形保存
├── ui/
│   ├── main_window.py   # メインウィンドウ
│   └── widgets.py       # トランスクリプト表示、デバイス選択等
└── config.json
```

## 7. config.json(初期案)

```json
{
  "model": "small",
  "compute_type": "int8",
  "device": "cpu",
  "language": "ja",
  "initial_prompt": "以下はオンライン会議の文字起こしです。",
  "vad": { "min_silence_ms": 600, "min_speech_ms": 250 },
  "output_dir": "C:/Users/.../Documents/transcripts",
  "follow_default_output": true,
  "mic_device_name": null,
  "always_on_top": false
}
```

`initial_prompt` に業務固有の用語(製品名・略語など)を書いておくと固有名詞の認識精度が上がる(F-10)。

## 8. 実装マイルストーン(Claude Codeへの依頼順)

各ステップで動作確認してから次へ進む。

1. **M1: 音声取得PoC**
   PyAudioWPatchでデバイス一覧表示 → マイクとWASAPIループバックの2系統を同時録音し、それぞれwav保存できることを確認(Bluetoothイヤホンで実会議相当のテスト)。
2. **M2: バッチ文字起こし・モデル比較**
   M1のwavを `small` と `kotoba-whisper-v2.0-faster`(いずれもint8/CPU)で文字起こしし、精度と処理速度(音声長に対する処理時間の比率)を比較して採用モデルを決定。比率が0.5を切っていればリアルタイム処理に十分な余裕あり。
3. **M3: ストリーミングパイプライン**
   silero-vad + セグメントキュー + ASRワーカーを組み、コンソールにリアルタイム出力。遅延と時系列マージを確認。
4. **M4: UI統合**
   PySide6でメインウィンドウ実装。開始/停止、デバイス選択、逐次表示、常に手前表示。
5. **M5: 出力・設定**
   逐次追記、停止時整形保存、config.json読み書き、保存先設定。
6. **M6: 堅牢化**
   Bluetooth切断/再接続ハンドリング、長時間稼働テスト(メモリ監視)、例外時のログ出力。

## 9. リスクと対策

| リスク | 対策 |
|---|---|
| ループバック対象と実際の再生デバイスの不一致で相手音声が無音 | 「既定デバイス追従」を既定ON。無音が30秒続いたらUIに警告表示 |
| Zoom+Whisper同時稼働でCPU逼迫 | smallモデル+int8を既定。ASRを1スレッドに制限。GPUがあればCUDAへ |
| Bluetooth切断でストリーム例外 | キャプチャスレッドで捕捉→再接続リトライ(指数バックオフ、最大5回) |
| 長い連続発話でセグメントが巨大化 | 最大セグメント長30秒で強制分割 |
| 会議アプリが排他モードで音声デバイスを占有 | 共有モードでのオープンを明示。エラー時はデバイス変更を促すメッセージ |
